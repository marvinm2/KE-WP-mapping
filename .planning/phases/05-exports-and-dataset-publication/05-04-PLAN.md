  <what-built>
  All Phase 5 export and publication features:
  - GMT exporter (KE-WP + KE-GO) with batch WikiPathways SPARQL and GO annotation lookup
  - RDF/Turtle exporter (rdflib-based, full Phase 2/3 provenance)
  - Zenodo uploader module (first-publish + new-version workflows)
  - Admin routes: POST /admin/exports/regenerate, POST /admin/exports/publish-zenodo
  - Public routes: GET /downloads, GET /exports/gmt/ke-wp, GET /exports/gmt/ke-go, GET /exports/rdf/ke-wp, GET /exports/rdf/ke-go
  - downloads.html page with 4 export cards
  - Stats page export download buttons
  - Navigation DOI badge (from zenodo_meta context processor)
  - data/zenodo_meta.json initial state file
  - ZENODO_API_TOKEN documented in .env.example
  </what-built>

  <how-to-verify>
  Start the dev server: `python app.py &`

  **Test 1 — Downloads page**
  1. Visit http://localhost:5000/downloads
  2. Confirm page loads (no 500 error)
  3. Confirm four export cards are visible (KE-WP GMT, KE-GO GMT, KE-WP Turtle, KE-GO Turtle)
  4. Confirm description text explains GMT format and HGNC symbols
  5. Confirm "Downloads" link is active in navbar

  **Test 2 — Stats page export buttons**
  1. Visit http://localhost:5000/stats
  2. Scroll to bottom — confirm "Export Formats" section with 4 download buttons is present
  3. Confirm buttons link to /exports/* paths

  **Test 3 — GMT download (on-demand generation)**
  1. Click "All mappings" under KE-WP GMT on /downloads (or visit http://localhost:5000/exports/gmt/ke-wp directly)
  2. A file should download. Check:
     - Filename format: `KE-WP_YYYY-MM-DD_All.gmt` (today's date)
     - File is tab-separated text (open in text editor)
     - First field of each row matches `KE{N}_{Slug}_{WP_ID}` format (e.g. `KE55_Decreased_BDNF_WP4655`)
     - Second field is a pathway name
     - Remaining fields are HGNC gene symbols (e.g. `CASP3`, `BCL2`)
     - No rows with only 2 columns (no empty-gene rows)
  3. If no KE-WP mappings exist in the DB yet, route should return 503 gracefully (not 500)

  **Test 4 — KE-GO GMT download**
  1. Visit http://localhost:5000/exports/gmt/ke-go
  2. File downloads with `KE-GO_YYYY-MM-DD_All.gmt` filename
  3. Rows have `KE{N}_{Slug}_{GO_ID}` term names (e.g. `KE55_Decreased_BDNF_GO:0006915`)
  4. Gene columns are HGNC symbols from GO annotations

  **Test 5 — RDF/Turtle downloads**
  1. Visit http://localhost:5000/exports/rdf/ke-wp — file downloads as `ke-wp-mappings.ttl`
  2. Open in text editor — confirm it starts with `@prefix` declarations and has `ke-wp:KeyEventPathwayMapping` triples
  3. Confirm UUID, confidence level, and (if available) curator and approval date are present
  4. Repeat for http://localhost:5000/exports/rdf/ke-go

  **Test 6 — Confidence filter**
  1. Visit http://localhost:5000/exports/gmt/ke-wp?min_confidence=High
  2. File downloads as `KE-WP_YYYY-MM-DD_High.gmt`
  3. All rows in the file should have been from High-confidence mappings only (verify by checking a mapping's confidence in the explore page)

  **Test 7 — Admin regenerate**
  1. Log in as admin and visit the admin panel
  2. In the browser dev console (or via curl), POST to http://localhost:5000/admin/exports/regenerate
     `curl -X POST http://localhost:5000/admin/exports/regenerate -H "Cookie: session=YOUR_SESSION"`
  3. Confirm response is JSON: `{"status": "ok", "files": [...], "message": "..."}`
  4. Confirm static/exports/ directory now contains the generated files

  **Test 8 — DOI badge in navbar**
  1. Manually edit `data/zenodo_meta.json` to set a test DOI:
     `{"deposition_id": 1234567, "doi": "10.5281/zenodo.1234567", "concept_doi": "10.5281/zenodo.9999999", "published_at": "2026-02-21"}`
  2. Restart the server: `pkill -f "python.*app.py" && python app.py &`
  3. Visit http://localhost:5000/ — confirm a "Cite: 10.5281/zenodo...." DOI link appears in the navbar header area
  4. Revert zenodo_meta.json to null state after verification

  **Test 9 — Navigation Downloads tab**
  1. Visit any page — confirm "Downloads" appears in the navigation bar
  2. Click it — confirm it navigates to /downloads

  Note: Zenodo API publish (Test for EXPRT-04) requires a real ZENODO_API_TOKEN. If not configured, the admin publish endpoint should return 503 with a clear message — verify this behavior rather than live publishing.
  </how-to-verify>

  <resume-signal>Type "approved" if all tests pass, or describe which tests failed with details of the failure (e.g., "Test 3 failed — GMT file has empty gene columns").</resume-signal>
